## Voice Orchestrator

The Voice Orchestrator is the central brain for every call. It is a dedicated microservice that sits between the telephony provider (Twilio) and the backend AI pipeline, coordinating the real-time audio → text → AI → text → audio loop for each call session.

**Receiving audio from telephony:** When a call arrives, Twilio opens a bidirectional Media Stream (WebSocket) and pipes the caller's raw audio to the Voice Orchestrator in small chunks. The Orchestrator maintains a per-call state machine that tracks which phase the call is in — greeting, listening, speaking, transferring, or ended.

**Streaming audio to ASR:** The raw audio chunks are forwarded in real time to the Speech-to-Text engine (Google Cloud Speech-to-Text or Azure Speech). The ASR returns intermediate (partial) transcriptions and a final transcription when the caller finishes speaking. The Orchestrator detects end-of-speech via silence thresholds, at which point the final transcript is considered complete.

**Sending transcript + context to the AI endpoint:** Once the transcript is finalised, the Orchestrator builds a context object:

```json
{
  "callId": "abc-123",
  "tenantId": "abc",
  "language": "ta",
  "transcript": "Fee payment deadline eppothu?",
  "previousTurns": [
    {"role": "ai", "text": "Vanakkam, ABC College help desk. How can I help you?"},
    {"role": "caller", "text": "Fee deadline pathi sollunga."}
  ]
}
```

It sends this payload to the backend's `POST /ai/answer` endpoint.

**Receiving the AI response and passing to TTS:** The endpoint returns an `AiResponse` object (see next section). If `shouldEscalate` is `false`, the Orchestrator sends `answerText` to the Text-to-Speech engine (Google Cloud TTS or Azure TTS) which synthesises natural speech in the caller's language. The audio stream is sent back through the Twilio Media Stream to the caller. If `shouldEscalate` is `true`, the Orchestrator plays a brief "connecting you to a staff member" message and initiates a warm transfer to a human agent.

## AI Response Layer

The AI response is generated by the **`POST /ai/answer`** endpoint on the `api-server`. This endpoint does NOT call the LLM directly — it first retrieves relevant context for the tenant, then constructs a prompt and sends it to the LLM.

**Input to the endpoint:**

```json
{
  "callId": "abc-123",
  "tenantId": "abc",
  "language": "ta",
  "transcript": "Fee payment deadline eppothu?",
  "previousTurns": [...]
}
```

**Processing pipeline inside the endpoint:**

1. **Retrieve tenant FAQs/documents** — The endpoint queries the database for the tenant's FAQ entries and document chunks relevant to the transcript (via embedding similarity search in a vector DB like pgvector or Pinecone). The top-k results are included as context.

2. **Build the LLM prompt** — The system prompt includes:
   - The tenant's identity and branding (e.g., "You are the AI assistant for ABC College.")
   - The retrieved FAQ/document snippets as grounding context
   - The conversation history (`previousTurns`)
   - Instructions to classify the issue category, rate confidence, and decide escalation
   - The instruction to respond in the caller's detected language (`language`)

3. **Call the LLM** — The assembled prompt + transcript is sent to the LLM (GPT-4 / Gemini). The LLM is instructed to return a structured JSON response.

**Output — the `AiResponse` object:**

```json
{
  "answerText": "The fee payment deadline for this semester is December 15th. You can pay online through the college portal or visit the accounts office.",
  "confidenceScore": 0.94,
  "issueCategory": "fees",
  "shouldEscalate": false,
  "language": "ta"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `answerText` | string | The AI's spoken response in the caller's language |
| `confidenceScore` | float (0.0–1.0) | How confident the AI is that the answer is correct and complete |
| `issueCategory` | string | Classified category: `"admission"`, `"fees"`, `"hostel"`, `"transport"`, `"placements"`, `"scholarships"`, `"office_hours"`, `"fee_payment"`, `"complaint"`, `"general"` |
| `shouldEscalate` | boolean | Whether this call should be handed off to human staff |
| `language` | string | Language code: `"en"`, `"hi"`, `"ta"`, `"te"`, `"kn"` |

The LLM is instructed to set `shouldEscalate` based on both the content of the query and its own confidence (see next section).

## Escalation Logic

The AI sets `shouldEscalate = true` under any of the following conditions. These rules are encoded in the LLM system prompt and reinforced with a post-processing check on the response:

| Condition | Rule | Example phrases |
|-----------|------|----------------|
| **Low confidence** | `confidenceScore < 0.7` — the AI could not find a relevant FAQ chunk or is uncertain about the answer. | Any ambiguous or out-of-scope query. |
| **Payment issues** | The transcript contains phrases related to payment failures, double charges, refunds, or transaction discrepancies. | "My payment failed but money was deducted", "I was charged twice", "Refund status" |
| **Emergency wording** | The transcript contains words like "emergency", "urgent", "accident", "medical", "harassment", "unsafe", "help me quickly". | "There's been an accident on campus", "This is urgent", "Medical emergency" |
| **Angry / complaint tone** | Sentiment analysis (built into the LLM prompt) detects anger, frustration, or the query is explicitly a formal complaint. | "This is ridiculous", "I want to file a complaint", repeated swearing or yelling |
| **Explicit staff request** | The caller directly asks to speak to a human. | "Connect me to someone", "I want to talk to the principal", "Get me a human" |

If `shouldEscalate` is `true`, the backend creates an `Escalation` record in the database with the call ID, escalation reason, and timestamp. This appears instantly in the admin dashboard's escalations queue. The admin can claim the escalation, review the transcript and recording, and mark it resolved.

## Integration with Backend API

The Voice Orchestrator and the backend `api-server` communicate via a single REST endpoint for AI processing, plus a logging endpoint for persisting results.

**1. Voice Orchestrator calls `POST /ai/answer`**

The Orchestrator sends the transcript, context, and tenant information to the `api-server`. This keeps the AI logic (RAG, prompt construction, LLM calls) within the backend, so the Orchestrator remains a lightweight streaming coordinator.

**2. `api-server` queries tenant-specific data**

On receiving the request, the `api-server`:
- Looks up the tenant's FAQ entries and embedded document chunks (scoped by `tenantId`)
- Retrieves any relevant snippets via vector similarity search
- Builds the LLM prompt with the tenant's context and the conversation history
- Calls the LLM (GPT-4 / Gemini) with the assembled prompt
- Parses the structured `AiResponse` from the LLM output

**3. `api-server` logs the AiResponse**

After receiving the AiResponse from the LLM, the backend persists it to the database:

- Updates the `Call` record with `issue_category`, `confidence_score`, `escalation_flag`, and `language`
- Stores the full transcript as a JSON blob
- Stores the `answerText` for later review
- If escalated, creates an `Escalation` record with the reason and callId

**4. Data appears in the admin Call Detail page**

The `web-admin` frontend fetches call data via `GET /calls/:callId` and displays:
- The transcript with speaker labels (Caller / AI)
- The AI-generated answer text
- The confidence score, issue category, and escalation flag
- A link to the call recording in object storage

This integration ensures that every AI interaction is auditable, searchable, and reviewable by college staff.

## Sequence of a Typical AI Answer

Below is the step-by-step sequence from the moment a caller stops speaking to the moment they hear the AI's answer:

```
 1. Caller finishes speaking
         │
 2. Voice Orchestrator detects end-of-speech (silence threshold)
         │
 3. ASR returns final transcript
         │
 4. Voice Orchestrator builds context object:
    { callId, tenantId, language, transcript, previousTurns }
         │
 5. Orchestrator sends POST /ai/answer with context
         │
 6. api-server receives the request
         │
 7. api-server queries vector DB for tenant's FAQs/docs
    (embedding similarity search scoped to tenantId)
         │
 8. api-server builds LLM prompt:
    [system prompt + tenant context + retrieved snippets + transcript]
         │
 9. api-server calls LLM (GPT-4 / Gemini)
         │
10. LLM returns structured AiResponse:
    { answerText, confidenceScore, issueCategory, shouldEscalate, language }
         │
11. api-server logs AiResponse to database:
    - Updates Call record (category, confidence, escalation flag, language)
    - Stores transcript and answerText
    - Creates Escalation record if shouldEscalate == true
         │
12. api-server returns AiResponse to Voice Orchestrator
         │
13. Voice Orchestrator checks shouldEscalate:
    │
    ├── false ──→ 14a. Send answerText to TTS engine
    │                     │
    │                    15a. TTS synthesises speech in caller's language
    │                     │
    │                    16a. Audio streamed via Twilio → Caller hears answer
    │
    └── true ───→ 14b. Play "connecting to staff" message
                        │
                       15b. Warm-transfer call to human agent
                        │
                       16b. Escalation visible in admin dashboard
```

## The LLM Provider Abstraction

The AI service depends on an **`LLMProvider` interface**, never on a specific vendor SDK. Claude, GPT-4, and Gemini are interchangeable adapters behind it; the provider is chosen by env/config (and can differ per tenant for cost/quality reasons). No provider SDK is imported outside `api-server/ai/` and `packages/prompts`.

```ts
// api-server/ai/provider.ts
export interface LLMProvider {
  complete(input: {
    system: string;
    messages: { role: "user" | "assistant"; content: string }[];
    responseFormat: "json";
  }): Promise<{ text: string; model: string; usage?: TokenUsage }>;
}
// adapters: ClaudeProvider, OpenAIProvider, GeminiProvider — all satisfy LLMProvider
```

**Structured output.** The LLM is instructed to return **strict JSON** matching `AiResponse`. The service parses it and validates with the `AiResponseSchema` from `packages/types`. If parsing/validation fails, it runs one bounded "repair" retry; if that also fails, it falls back to a **safe escalation** response rather than passing malformed data downstream.

**Prompt assembly** pulls templates from `packages/prompts`, parameterised by `language` and (once classified) `issueCategory`, and injects: tenant identity/branding, retrieved FAQ/document snippets (with provenance), the conversation history, and the escalation rules. Keeping prompts in their own package lets Tanishqa & Surya iterate on prompt quality without touching application code.

## Failure Handling

The AI layer's guiding rule: **fail safe, not silent.** Every failure mode resolves to a human hand-off or a bounded retry — never a confident wrong answer.

| Failure | Handling |
|---|---|
| LLM returns invalid/non-JSON | One repair retry → else safe-escalation `AiResponse` (`shouldEscalate: true`). |
| LLM confidence `< 0.7` | Deterministic post-check forces `shouldEscalate: true` regardless of what the model said. |
| Retrieval finds no relevant chunk | Treated as low confidence → escalate; the gap is surfaced to admins as an FAQ to add. |
| LLM / ASR / TTS timeout or 5xx | Bounded retry with timeout; on exhaustion, `/ai/answer` returns escalation (or `502 UPSTREAM_UNAVAILABLE`). |
| Prompt-injection / off-topic input | System prompt constrains scope to the tenant's domain; out-of-scope → escalate or safe refusal. |

The deterministic post-check means escalation is **not** solely the model's judgement — the rules in the Escalation Logic table above are re-applied in code after the LLM responds.
